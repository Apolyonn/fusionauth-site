Writing guidelines

Thanks for writing for us. Here are some guidelines to help make the process go smoothly.

  • We typically look for blog content. You can check the blog out to get a feel. We have a slightly irreverant tone and enjoy posts that tell stories. While we want FusionAuth to play a role in any writing, it doesn't have to be the star. We've found that posts that focus on helping a user with a more general problem like authentication, JWTs or authorization are better recieved than a 'how do I do this with FusionAuth' post.
  • Our audience is developers and managers of developers. We want the content to be technical in nature. We have an ongoing list, but are also happy to hear pitches based on your skillset and interests. Some sample ideas:
    • How to use OAuth to secure an application written in [language].
    • Securing [application or framework] with OAuth.
    • How to use [specific OAuth grant] in [language].
    • Can you run FusionAuth in Fargate/GKE/Heroku? How?
    • What's next for OAuth and OIDC?
    • How I can set up single sign on for [SaaS application], using FusionAuth as the identity provider?
    • What is the difference between ECC and RSA public key cryptography?
    • What is PKCE and why would you use it?
  • The typical process is:
    1. Contact is made. Please show us an example of your writing.
    2. We discuss a few topics based on our needs and your interests.
    3. We agree on a topic.
    4. You send over an outline.
    5. We give feedback and we discuss the outline, and then I approve it via email.
    6. You write the article.
    7. You deliver it.
    8. We give feedback or ask questions after review. Typically we do 1-2 revisions.
    9. I approve it.
    10. I get payment scheduled.
    11. We publish the article.
  • Additional guidelines:
    • Please provide your paypal email address, as that's how we'll pay you.
    • Please provide your twitter handle, and we'll tag you when we publish the post.
    • If you'd like to republish the article on your own blog or medium site, you may do so. We ask that you wait a week after we publish it and that you link back to the FusionAuth blog as the location of the original publication.
    • The FusionAuth site is open source and uses jekyll so you can run it locally if you'd like.
    • Indent all code blocks with two spaces per level.
    • Use the authorization code grant if you are working with OAuth (instead of the implicit grant). Please use the "state" parameter to defend against CSRF attacks and PCKE for additional security.
    • Make sure you store all tokens securely, as outlined here.
    • If you make an example application, we'd love to have that transferred over to our github repository, where we'll make it available under the Apache license. If you'd like to add yourself as a contributor in the project readme, please feel free to do so.
    • We prefer to have the article delivered in jekyll compatible markdown. If that doesn't work, then google docs is fine, but we still need the images separately.
    • When you provide images or videos, please make them available as a zip file. We post-process images included in our posts.
    • If you take screenshots of the FusionAuth interface, please don't spend a lot of time on making them look good. We re-do them in house with a specific host configuration (using Silicon Vally characters). All you to do is take a screen shot with enough detail that we can recreate it.
    • If you make a video of any part of the process, please give us the source file so we can upload it to our youtube channel.
    • We'll notify you when the article is published, and would appreciate any social sharing you can do.
    • Please run your article through Google docs grammar and spell checker and correct any errors before submitting it.
